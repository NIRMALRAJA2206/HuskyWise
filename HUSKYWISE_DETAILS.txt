================================================================================
          HUSKYWISE: IN-DEPTH SYSTEM ARCHITECTURE & WORKFLOW
================================================================================

1. OVERVIEW
-----------
Huskywise is a Retrieval-Augmented Generation (RAG) application built with 
Next.js. It allows users to upload documents (PDF, DOCX, TXT, MD), index them 
using local or cloud-based AI models, and then query those documents using 
natural language. The system combines precise document parsing with semantic 
search to provide grounded, context-aware answers.

2. CORE TECHNOLOGIES
--------------------
* Frontend/Backend: Next.js (TypeScript)
* AI Orchestration: Ollama (Local) or OpenAI (Cloud)
* Document Parsing: pdf-parse (PDFs), mammoth (DOCX)
* Vector Ops: Manual Cosine Similarity (lib/retrieval.ts)
* Storage: Filesystem-based JSON Store (.huskywise/index.json)

3. THE DATA INGESTION PIPELINE (STEP-BY-STEP)
---------------------------------------------

Step 1: Parsing (lib/parsers.ts)
- When a file is uploaded, the system detects its extension.
- PDF: Uses `pdf-parse` with a dedicated worker to extract text page-by-page.
- DOCX: Uses `mammoth` to convert Office XML into raw text.
- Text/MD: Decoded directly from binary using `TextDecoder`.

Step 2: Chunking (lib/chunking.ts)
- Extracted text is broken down into smaller "chunks" for better retrieval.
- Configuration:
  * MAX_CHARS: 1100 (target size per chunk)
  * OVERLAP_CHARS: 180 (ensures context continuity between chunks)
- Intelligent Splitting: The algorithm attempts to split at sentence 
  boundaries (. ) or newlines (\n) to avoid cutting words in half.

Step 3: Embedding (lib/llm.ts)
- Each text chunk is sent to an embedding model (e.g., `nomic-embed-text`).
- The model converts the text into a high-dimensional vector (an array of 
  numbers) that represents its semantic meaning.

Step 4: Storage (lib/store.ts)
- The resulting vectors, along with the original text and metadata (page 
  numbers, document IDs), are saved into `.huskywise/index.json`.
- This local "vector database" allows for fast retrieval without re-parsing.

4. THE RETRIEVAL-AUGMENTED GENERATION (RAG) FLOW
-------------------------------------------------

Phase A: Retrieval (lib/retrieval.ts)
1. Query Embedding: The user's question is converted into a vector using 
   the same embedding model used during ingestion.
2. Semantic Search: The system calculates the 'Cosine Similarity' between 
   the query vector and every chunk in the store.
3. Top-K Ranking: The 8 most relevant chunks (highest similarity scores) 
   are selected as context.

Phase B: Augmentation & Generation (app/api/chat/route.ts)
1. Prompt Engineering: The system constructs a prompt that includes the 
   8 retrieved chunks and the user's original question.
2. Grounding: The LLM (e.g., Llama 3.1) is instructed to answer *only* 
   using the provided context.
3. Final Answer: The LLM generates a response that is accurate, cited (if 
   implemented), and grounded in the uploaded documents.

5. ERROR HANDLING & RESILIENCE
------------------------------
* Retry Logic: (lib/llm.ts) implements a `withRetry` wrapper for API 
  calls to handle transient network or model-loading failures.
* Resource Management: PDF parsers are explicitly destroyed after use 
  to prevent memory leaks in the Node.js environment.
* Fake Worker Fix: The system manually resolves paths to `pdf.worker.mjs` 
  to ensure compatibility with Next.js Turbopack.

6. SUMMARY OF USER BENEFITS
---------------------------
- Privacy: Can run 100% locally using Ollama.
- Speed: In-memory/local-file storage eliminates database latency.
- Accuracy: RAG reduces "hallucinations" by forcing the AI to look 
  at the source text before speaking.
